Unordered Lists

An unordered list displays items with bullet points. Order doesn't matter — a grocery list, a set of features, a navigation menu. Use <ul> for the list and <li> for each item:

unordered list 
<h3>Skills I'm Learning</h3>
<ul>
  <li>HTML</li>
  <li>CSS</li>
  <li>JavaScript</li>
  <li>React</li>
</ul>

What the browser shows

Skills I'm Learning

  • HTML
  • CSS
  • JavaScript
  • React

Ordered Lists

An ordered list displays items with numbers. Use <ol> when sequence matters — steps in a recipe, ranking, instructions:

ordered list 
<h3>How to Make Coffee</h3>
<ol>
  <li>Boil water</li>
  <li>Grind the beans</li>
  <li>Pour water over grounds</li>
  <li>Wait 4 minutes</li>
  <li>Enjoy!</li>
</ol>

You can customize the starting number and even the numbering type:

ordered list options 
<!-- Start counting from 5 -->
<ol start="5">
  <li>This is item 5</li>
  <li>This is item 6</li>
</ol>

<!-- Use letters instead of numbers -->
<ol type="A">
  <li>First (A)</li>
  <li>Second (B)</li>
</ol>

<!-- Roman numerals -->
<ol type="I">
  <li>Chapter I</li>
  <li>Chapter II</li>
</ol>

Nested Lists

Lists can contain other lists. This is super useful for sub-items, categories, and multi-level navigation:

nested lists 
<ul>
  <li>Frontend
    <ul>
      <li>HTML</li>
      <li>CSS</li>
      <li>JavaScript</li>
    </ul>
  </li>
  <li>Backend
    <ul>
      <li>Node.js</li>
      <li>Python</li>
    </ul>
  </li>
</ul>

The nested <ul> must go inside the parent <li>, not after it. A common mistake is closing the <li> before adding the nested list. The inner list is content of that list item.

Description Lists

There's a third type of list most beginners don't know about: the description list. It pairs terms with their definitions:

description list 
<dl>
  <dt>HTML</dt>
  <dd>HyperText Markup Language — the structure of web pages.</dd>

  <dt>CSS</dt>
  <dd>Cascading Style Sheets — the visual styling of web pages.</dd>

  <dt>JS</dt>
  <dd>JavaScript — the behavior and interactivity of web pages.</dd>
</dl>

<dl> is the list container, <dt> is the term, and <dd> is the description. Great for glossaries, FAQs, and metadata displays.

Tables: The Basics

Tables organize data into rows and columns. They're built with a handful of tags that you'll pick up quickly:

basic table 
<table>
  <tr>
    <td>Row 1, Cell 1</td>
    <td>Row 1, Cell 2</td>
    <td>Row 1, Cell 3</td>
  </tr>
  <tr>
    <td>Row 2, Cell 1</td>
    <td>Row 2, Cell 2</td>
    <td>Row 2, Cell 3</td>
  </tr>
</table>

The building blocks:

  • <table> — The table container
  • <tr> — Table Row (one horizontal row)
  • <td> — Table Data (one cell)

Tables: Headers, Captions & Structure

A proper, accessible table uses header cells and structural sections:

well-structured table 
<table>
  <caption>Q1 Sales Results</caption>

  <thead>
    <tr>
      <th>Month</th>
      <th>Revenue</th>
      <th>Growth</th>
    </tr>
  </thead>

  <tbody>
    <tr>
      <td>January</td>
      <td>$12,000</td>
      <td>+5%</td>
    </tr>
    <tr>
      <td>February</td>
      <td>$14,500</td>
      <td>+20%</td>
    </tr>
    <tr>
      <td>March</td>
      <td>$13,800</td>
      <td>-5%</td>
    </tr>
  </tbody>

  <tfoot>
    <tr>
      <td>Total</td>
      <td>$40,300</td>
      <td>—</td>
    </tr>
  </tfoot>
</table>

Here's what each new element does:

  • <caption> — A title/description for the table (great for accessibility)
  • <thead> — Groups the header row(s)
  • <tbody> — Groups the body data rows
  • <tfoot> — Groups the footer row(s) (totals, summaries)
  • <th> — Table Header cell (bold and centered by default; tells screen readers this is a header)

Always use <th> for header cells instead of bolding regular <td> cells. Screen readers announce <th> as headers, which helps visually impaired users understand the table structure.

When NOT to Use Tables

This is important: tables are for data, not for page layout.

Back in the early days of the web (late '90s, early 2000s), developers used tables to create multi-column page layouts. This is now considered a terrible practice because it creates inaccessible, fragile, hard-to-maintain code. CSS handles layout now — specifically Flexbox and Grid.

Use a table when you have tabular data: schedules, comparison charts, stats, pricing. Don't use it to arrange your navigation, sidebar, and content area.

A good test: if your content would make sense in a spreadsheet, it belongs in a table. If not, use a different HTML structure and style it with CSS.
